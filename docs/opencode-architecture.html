Swapping the execution agent: Claude Code ⇄ OpenCode

An exploratory architecture for making the code-execution agent a per-project choice — Anthropic's Claude Code today, OpenCode as a second backend that opens the door to any provider OpenCode can reach (OpenAI, Google, Amazon Bedrock, OpenRouter, local Ollama, …). The goal is twofold and weighted equally: stop hitting Claude usage limits in our own workflow, and let people who cannot use Claude Code at all (Bedrock-only shops, no-Anthropic-subscription users) run Sandstorm anyway.

⟵ system architecture  ·  telemetry architecture

The headline: the seam already exists. src/main/agent/types.ts defines an AgentBackend interface and ClaudeBackend is its single implementation — the original design anticipated other providers. This work is therefore "add a second implementation + a selection layer," not "untangle Claude from everything." And OpenCode's primitives line up with Claude Code's almost 1:1, so most of the existing machinery — the dual-loop harness, the MCP orchestration bridge, the telemetry pipeline — is reused rather than rewritten.

§ 1 The seam that already exists

Claude Code runs in exactly two places, and both already pass through abstractable boundaries. One per-project setting routes both to the chosen backend.

The outer orchestrator is the chat session that manages stacks (the one that burns tokens juggling). It talks to ClaudeBackend through the AgentBackend interface in src/main/agent/types.ts. The inner worker is Claude running inside each stack's Docker container, invoked by sandstorm-cli/docker/task-runner.sh, which already parses Claude's stream-json into the events its dual-loop harness consumes. Both boundaries are the natural place to branch on a backend choice. The choice itself lives next to today's model settings in the SQLite registry (src/main/control-plane/registry.ts), set from a new section of the project config modal — the only user-visible change.

§ 2 Inner worker — one image, env-selected runner

The inner worker is where most token volume lives, so it matters most. The recommendation: ship both CLIs in the one existing Docker image and branch at runtime.

task-runner.sh already wraps Claude in a run_claude() function that runs claude --dangerously-skip-permissions --output-format stream-json and pipes the NDJSON through jq to extract text and tool calls. OpenCode's headless mode is a near-mirror: opencode run "<prompt>" --format json --dangerously-skip-permissions emits one JSON object per line with a type field (text, tool_use, step_start, step_finish, error). The integration is a sibling run_opencode() that parses that stream into the same normalized { text, tool, status } shape the harness already consumes. Everything above that line — execution ↔ review ↔ verify, meta-review, the /tmp/claude-task.* status files — is reused unchanged (the status-file names become a thin backend-neutral contract; renaming is cosmetic and optional).

Why one image, not two

  • One image · env-selected runner (recommended). Add the OpenCode CLI to the existing sandstorm-cli/docker/Dockerfile alongside claude. The entrypoint and task-runner.sh branch on an AGENT_BACKEND env var. One image to build and cache; the CLIs are small; switching backend per stack costs nothing.
  • Per-backend images (rejected). A separate opencode image means image proliferation, doubled build/cache time, and two Dockerfiles to keep in lockstep. No upside when both CLIs coexist happily in one container.
The one real divergence to design for is prompt delivery. Claude takes the task prompt on stdin today; OpenCode's headless run has open bugs around stdin (it can hang). The safe, backend-neutral contract is to write the prompt to a temp file and pass it as an argv positional / --file for OpenCode, keeping stdin for Claude. Both already read the prompt from /tmp/claude-task-prompt.txt on the host side, so this is a one-branch change in the runner.

§ 3 Outer orchestrator — OpenCode server mode

For the chat session, drive OpenCode through its server + SDK, not by spawning a one-shot per turn. This mirrors how ClaudeBackend keeps a long-lived process and streams events back to the renderer.

OpenCode is built as a TUI client talking to a backend HTTP server, and that server runs standalone: opencode serve exposes an OpenAPI REST surface plus a Server-Sent-Events stream at /event, and there's an official @opencode-ai/sdk generated from the spec. OpenCodeBackend implements the existing AgentBackend interface by mapping each method onto the SDK: sendMessage → create/continue a session and session.prompt(); streaming chunks → subscribe to /event; runEphemeralAgent / spawnEphemeralAgent → a short-lived session or a one-shot opencode run --format json; getAuthStatus / login → opencode auth + provider config. The renderer and the IPC bridge don't change — they already speak only to the interface.

Renderer chat UI  --IPC-->  preload bridge  -->  AgentBackend interface

        sendMessage(tabId, msg)   -> session.create / session.prompt   (REST)

§ 4 Instructions & config injection — the genuinely different part

Two things actually differ between the backends: where project instructions go (today's injected inner CLAUDE.md) and how project settings reach the agent. Both resolve cleanly to "one composed artifact, two delivery mechanisms."

Today the control plane composes the inner agent's instructions and writes them as CLAUDE.md (mounted at ~/.claude/CLAUDE.md in the container, with CLAUDE_CODE_DISABLE_CLAUDE_MDS controlling auto-injection). OpenCode reads project instructions from AGENTS.md and from an instructions: [...] array in opencode.json (globs and URLs allowed) — and, helpfully, it reads CLAUDE.md as a fallback when no AGENTS.md is present. So the same composed file feeds both backends: Claude reads it natively; OpenCode either picks it up via the fallback or, more deterministically, via an explicit instructions entry pointing at it.

Project settings reach OpenCode through a generated opencode.json injected non-invasively via the OPENCODE_CONFIG / OPENCODE_CONFIG_CONTENT env var (no need to write into the user's workspace). That generated config carries: the model as a provider/model string, provider credentials as {env:VAR} references, the Sandstorm orchestration MCP bridge as a type:"local" server (the existing stdio bridge, reused), an autonomous permission: "allow", and the instructions array. Credentials are supplied as container env vars — never baked into the image.

control plane composes instructions (existing, provider-neutral)

§ 5 Providers, credentials & autonomy

This is the payoff: with OpenCode selected, the model field becomes a provider/model string and credentials are whatever that provider needs. Bedrock and Ollama — the two most-requested "we can't use Claude Code" cases — are first-class.

ProviderHow model is namedCredential delivery (container env / config)
Anthropicanthropic/claude-…OAuth (existing sync) or ANTHROPIC_API_KEY
OpenAIopenai/gpt-…OPENAI_API_KEY
Amazon Bedrockamazon-bedrock/…AWS chain: AWS_BEARER_TOKEN_BEDROCK or AWS_PROFILE / access keys + region
Local Ollamacustom provider, OpenAI-compatiblebase URL http://host:11434/v1 — no key
OpenRouter / othersopenrouter/…provider apiKey via {env:VAR}

Autonomy. Inside the sandboxed container we want zero permission prompts, exactly as Claude runs today. OpenCode offers the same escape hatch two ways: opencode run --dangerously-skip-permissions (per run) or "permission": "allow" in the generated config. We use the config form so it also covers the server-driven outer path.

Credential precedence trap. A stored OpenCode auth.json OAuth token can silently override keys injected via config (known upstream issue). In-container we start from a clean auth.json and inject via env vars so the active credential is deterministic and matches what the user configured in the modal.

§ 6 Telemetry — stays on the rails

The telemetry rebuild already points at ccusage, and ccusage has first-class OpenCode support — so the backend swap is aligned with, not a detour from, where telemetry is headed.

OpenCode's --format json emits a step_finish event per step carrying cost and tokens.{input,output,reasoning,cache}; the same data is on message objects via the server API. A parent process sums these for per-run telemetry exactly like the current Claude path reads token usage off the result event. For the ccusage direction, OpenCode writes its own session data that ccusage already knows how to read, so mounting the inner agent's data dir generalizes the existing "mount ~/.claude/projects" approach to "mount whichever agent's session dir is in play."

§ 7 Configuration surface — the only thing users see

Per the design intent: no functional UX change. The app looks and behaves identically. The single addition is a section in the project config modal.

The modal gains an Agent backend selector (Claude Code / OpenCode). Choosing OpenCode reveals provider, model, and credential fields. These persist to the registry next to today's inner_model / outer_model settings — same storage, same per-project + global-default pattern, same getEffectiveModels-style resolution (now getEffectiveBackend). Nothing else in the renderer changes: the chat, the Kanban, the stack detail views all keep talking to the same interfaces. Sensible defaults mean a user who never opens the new section keeps running Claude Code exactly as before.

The honest list — what to verify empirically and decide before building.

Repo moved sst/opencode → anomalyco/opencode; fast CLI churn high Pin a version in the image; treat the NDJSON field names (part.tokens.*, part.cost) as needing empirical confirmation against the pinned build. Re-verify flags against opencode --help.

§ 10 Suggested rollout

Sequenced so value lands early and the risky/novel parts come after the seam is proven.

  1. Backend factory + registry field. Introduce backendType resolution and a backendFactory(project); no behavior change yet (Claude stays the only impl). Pure refactor with tests.
  2. Inner runner branch. Add opencode to the image and a run_opencode() that normalizes NDJSON into the harness's event shape. Gate behind AGENT_BACKEND. Verify a real task end-to-end in a stack.
  3. Config + instruction injection. Generate opencode.json + the instructions wiring; reuse the composed artifact and the MCP bridge.
  4. Provider/credential plumbing. Anthropic-via-OpenCode first (lowest variance), then Bedrock, then Ollama. Each is a credential-mapping + a smoke test.
  5. OpenCodeBackend (outer). Implement the interface over opencode serve + SDK. This is the largest single piece; it lands last because inner-worker value is already shipped.
  6. Config modal section + telemetry. Expose the selector; wire step_finish usage into the telemetry pipeline.
Bottom line. The architecture was built for this moment — a single interface, a single inner-runner seam, a single config artifact. Adding OpenCode is additive and reversible: every project defaults to Claude Code, and the new path is opt-in per project. The reward is escaping Claude usage limits for power users and unlocking Sandstorm for everyone who can only bring Bedrock, a local model, or some other provider.

§ G Glossary

AgentBackend
The existing interface (src/main/agent/types.ts) every execution agent implements. The whole swap hangs off this seam.
backendFactory
Proposed selector that returns ClaudeBackend or OpenCodeBackend for a project, read from the registry.
AGENT_BACKEND
Proposed container env var read by task-runner.sh to choose the inner runner.
run_opencode()
Proposed sibling of run_claude() that parses OpenCode NDJSON into the harness's normalized events.
opencode serve
OpenCode's standalone HTTP server (REST + /event SSE); the basis for the outer orchestrator path.
instructions[]
OpenCode config array listing instruction files (globs/URLs ok); OpenCode's analog to injected CLAUDE.md.
AGENTS.md
OpenCode's native project-instructions file; it falls back to CLAUDE.md when absent.
step_finish
OpenCode NDJSON event carrying per-step cost and token counts — the telemetry hook.
permission: "allow"
OpenCode config for fully autonomous tool use; the container-only analog of --dangerously-skip-permissions.
