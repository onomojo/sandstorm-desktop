§ 1 The problem

We lost token-usage visibility in the Kanban migration, and the telemetry we did have never worked reliably — the tree carries four-plus overlapping, mostly-broken attempts (§ 6). Meanwhile the ecosystem already solved this: Claude Code writes a complete, per-message usage record to local JSONL transcripts, and off-the-shelf tools read those to produce exactly the breakdowns we want.

Thesis. Stop hand-rolling telemetry. Adopt a proven reader as the aggregation engine, solve the one problem that is genuinely ours — getting inner-stack transcripts where we can read them and attributing them to tickets — and delete the rest.

Goals

  • Month / 30-day / 7-day cost and token totals, trending.
  • Daily time series — the "how much did I spend each day" graph.
  • Per-ticket cost (the headline ask) and per-model cost.
  • Input / output / cache-create / cache-read breakdown.
  • Drill into an individual session.
  • Attribute inner-stack usage, not just the outer orchestrator.

Non-goals (v1)

  • Real-time per-token meters — we already have a live session counter.
  • Hard budget caps / enforcement — a fast-follow once the data is trusted.
  • Replacing the /usage session-limit meter on day one (it can coexist — § 6).

§ 2 Tool landscape

The brief described a CLI that breaks down a month of usage with per-session drill-down. That is almost certainly ccusage — and it ships a library API, so we embed it rather than scrape stdout.

ToolWhat it isFit
ccusage
ryoppippi/ccusage		 Local CLI and library. Reads ~/.claude/projects/**/*.jsonl. Daily / weekly / monthly / session / 5-hour-block reports. Separates cache-create vs cache-read. Estimates USD from bundled pricing (offline). Groups by project. JS API: loadDailyUsageData(), loadSessionData(), … Recommended engine. Library mode = embed, don't shell out.
better-ccusage Community fork of ccusage. Fallback if upstream stalls.
claude-code-usage-analyzer Wraps ccusage + LiteLLM pricing for deeper cost analysis. Inspiration for cost modeling; heavier dependency.
Hand-rolled token-telemetry.ts Our own per-turn JSONL writer, opt-in via env var. Retire (§ 6).

Why ccusage as a library, not a CLI subprocess: no PTY/stdout parsing (the exact fragility that sank account-usage.ts); typed results we can cache and join to our own ticket/stack tables; the community-trusted pricing source so we don't maintain a price list; offline, nothing leaves the machine.

External-contract risk. Pin the ccusage version and treat the JSONL transcript schema as an external contract — Claude Code's format can change. A thin adapter in src/main/telemetry/ isolates that risk.

§ 3 The one hard problem: reading inner-stack transcripts

The outer orchestrator runs on the host, so its transcripts already sit in the host ~/.claude/projects/ that ccusage reads natively. Inner Claude runs in a Docker container, and today its ~/.claude is not visible to the host. The trick is to expose only the output transcripts — never the container's config.

From the current runtime setup (confirm during implementation): the stack's Claude home is /home/claude/.claude/; volume mounts today are the workspace, read-only context, and the docker socket — not the Claude home; and the entrypoint writes inner guidance into /home/claude/.claude/CLAUDE.md. That guidance is load-bearing — stacks work because of it.

Decision — mount only the projects/ subdirectory, read back to the host

Alternatives considered (§ 7): docker cp on task completion (an export step that can miss in-progress data); or keep the in-container token-counter.sh and persist only totals (loses the per-session granularity we want). Sub-decision to confirm: bind-mount vs named volume, plus the uid/permission model so the container writes and the host reads.

§ 4 Recommended architecture

ccusage reads every transcript root (host + mounted stacks); a thin telemetry adapter attributes each session to a ticket and caches rollups in SQLite; IPC feeds the renderer. All new code is host-side and sits behind a small interface — honoring the pluggable-agent-backend rule (ccusage is Claude-shaped; a future backend supplies its own usage adapter).

§ 7 Options & trade-offs

Four shapes considered. The recommendation is effectively D — ccusage for the bulk, plus our phase writer as the one first-party enrichment.

A · ccusage library + mount projects/  (recommended core)

➕ Proven engine, cost included, no parsing, persists past teardown, least new code. ➖ External JSONL-schema dependency (mitigated by an adapter + pinned version); one compose change per stack.

B · ccusage CLI via docker cp on task end

➕ No long-lived mount. ➖ Export can miss in-progress/aborted sessions; subprocess + JSON plumbing; nothing during a running task.

C · First-party pipeline (finish our own SQLite telemetry)

➕ Full control, no external dep, phase-aware natively. ➖ We re-implement pricing and aggregation — the exact work that has failed repeatedly here; most maintenance for least differentiation.

D · Hybrid  (the recommendation)

ccusage for cost/usage + our task_token_steps for the phase split. ➕ Best data — ccusage's rigor plus our dual-loop dimension. ➖ Two sources to reconcile.

§ 8 Rough phasing

Each phase is independently shippable and reviewable.

  1. Ingest the host (orchestrator) only. Add ccusage + adapter + stats:telemetry:* IPC. Ship the Telemetry view (mockup 01) reading real host data. No compose changes — immediate value, low risk.
  2. Attribute + persist. Join sessions → tickets; add the SQLite rollup cache. Light up per-ticket / per-model (mockups 02, 04, 05).
  3. Reach into stacks. Mount projects/ per stack (§ 3); union the roots. Inner-stack and per-ticket numbers become real and survive teardown.
  4. Board integrations. Add the chosen board surface(s) from 06–10.
  5. Cleanup. Remove dead telemetry (§ 6); decide task_token_steps; settle the /usage meter's future.

§ 9 Open questions

  • ccusage multi-root discovery: a native flag vs adapter-side glob + merge?
  • Mount mechanism: bind-mount vs named volume; uid / permission model.
  • Keep task_token_steps (phase split) or cut it? — drives whether we touch the dual-loop runner.
  • Does the existing /usage session-limit meter stay, or fold into the new telemetry once trusted?
  • Cost display: list price only, or account for plan/subscription? (ccusage estimates list price — fine for trend; label it "estimated".)

§ — Glossary

ccusage
Local CLI + library that reads Claude session transcripts and reports usage and estimated cost by day/month/session/model. The proposed read & cost engine.
Transcript
The per-session *.jsonl Claude Code writes under ~/.claude/projects/<project>/<session>.jsonl — one line per message, with token usage.
Attribution
Mapping a session's usage to a ticket, via tasks.session_id (preferred) or the stack's mounted usage dir; null ticket ⇒ outer/orchestrator.
Rollup cache
SQLite-stored pre-aggregated daily / per-ticket / per-model totals so the dashboard is instant and history outlives a torn-down stack.
Phase split
Execution / review / verify cost breakdown from the dual-loop — first-party, via task_token_steps. Not knowable to ccusage.
cache-create / cache-read
Prompt-cache write vs hit tokens, priced and counted separately. Cache-read dominates the mix.
